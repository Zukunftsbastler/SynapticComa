# Artwork Pipeline Roadmap: From Procedural Placeholders to a Generative, Narrative-True Look

**Status:** Concept only — nothing in this document is implemented yet.
**Trigger:** Till wants to replace the current placeholder-flat rendering with real artwork that fits the narrative (dual-dimension asymmetry, the Id vs the Superego), stays legible for a text-free, tile-based logic game, and is produced mostly through a local, scriptable generative-AI pipeline.

## 1. The Core Tension: Tile Grid vs. Organic Bleed — and How to Resolve It

Till's brief flags this directly: the game is mechanically tile-based (a hex grid is the whole logic substrate — the solver, routing, and collision all operate on discrete hexes), but the *theme* (a fracturing mind, tissue vs. circuitry) wants continuity, not a board-game grid of visibly separate tiles.

> **Decision reversed same day (2026-07-21) — read this before the rest of the section.** The original recommendation below (one continuous panorama image per dimension, cropped under the hex grid) was built and tested — see §4.1's history — but Till pushed back with a concrete objection: the generated backgrounds are square/rectangular, while tiles are hexagonal, and a single fixed-frame background can't support board shapes other than a disk-shaped radius later. His stated preference: **per-hex-tile sprites that together compose the board's shape**, so irregular/non-disk level layouts stay possible without ever needing to match a background frame. This is also, on reflection, the better technical fit for how this engine already renders hexes: `RenderSystem.ts`'s existing `drawHex` already draws every tile with a small inset gap between neighbors (a mosaic/chunky-diorama look, matching `art_and_ui.md §1`'s "thick, chunky tiles" language *literally*, not the continuous-painting reading this section originally argued for) — so tiles never needed to seam-match pixel-perfectly in the first place; each hex is already visually its own piece. **Current approach: floor tiles (`HEX_ID_FLOOR`/`HEX_SUPEREGO_FLOOR`) are ordinary per-entity sprites, exactly like any hazard or avatar icon** — see §3/§5, both now implemented. The single-panorama code and assets built for the paragraph below were removed the same day.

The rest of this section is kept for the record of the reasoning that was tried and superseded, since the underlying problem (would per-tile generation look like a repeated stamp?) is still worth understanding:

Generative image models are fundamentally bad at the thing a literal "seamlessly edge-matched organic tile set" would need — that's still true. But it turns out this game's rendering doesn't need that: because tiles already render with a visible gap/border between them (not a borderless continuous surface), each tile only needs to look good and thematically consistent on its own, not pixel-match its neighbors at the edge. A handful of tile variants per dimension (to avoid one exact image repeating obviously across many hexes) is the real remaining concern, not seam-matching — noted as a follow-up in Phase 1/2, not solved yet (only one tile texture exists per dimension so far).

**Foreground game pieces** (avatars, hazards, conduit plates, ability nodes, walls, exits) follow the same per-entity-sprite path as floor tiles — they stay legible, high-contrast icons, because the game's own design philosophy is absolute on this point (`architecture.md §3`: "every active component must have a direct visual representation," language-agnostic, no text anywhere). Atmosphere must never cost a player the ability to read the board at a glance.

## 3. The Engineering Prerequisite — done, 2026-07-21

Built the real texture pipeline `src/rendering/` needed before any generated image can appear in the game:
- **`src/registry/TextureRegistry.ts`** (new): a tiny non-PixiJS module tracking which `SpriteId`s currently have a loaded texture (`isSpriteLoaded`/`markSpriteLoaded`). Exists so `RenderSystem.ts` (pure ECS logic, no PixiJS import) can decide sprite-vs-procedural-fallback per entity without depending on `pixi.js` at all — preserves the existing architectural separation between game logic and the PixiJS adapter.
- **`RenderCommandBuffer.ts`**: `DrawSpriteCommand` (a pre-existing but unused/vestigial type from a long-abandoned "Sprint 9" attempt) extended with `width`/`height`/`alpha`.
- **`PixiDriver.ts`**: `preloadTextures()` runs once at construction, attempting `Assets.load()` for every path in `SPRITE_PATHS`; missing files (the normal case for anything not yet promoted) are caught and silently skipped — this must never block startup. A pooled `Sprite` array (mirroring the existing `textPool` pattern for `Text`) renders `drawSprite` commands. **Known simplification:** pooled sprites all render in one `spriteLayer` above the procedural `gfx` Graphics batch — true per-command z-interleaving between Graphics-drawn hexes and Sprites isn't practical with PixiJS's immediate-mode Graphics API. Irrelevant today (nothing stacks yet) but revisit if icon-over-icon layering ever matters once more assets are promoted (Phase 2+).
- **`RenderSystem.ts`**: each of the three rendering passes (floor tiles, entity hexes, avatars) now checks `isSpriteLoaded(sid)` first and emits `drawSprite` instead of the procedural `drawHex`/`drawCircle` fallback when a real texture exists — **incremental, per-`SpriteId`**, so art ships gradually without a big-bang cutover. Verified byte-for-byte behavior-preserving for every `SpriteId` with no promoted asset (i.e., everything except the two floor tiles as of this writing).
- **Floor tiles are ordinary per-entity sprites** — see §1's reversal above. No separate background-layer mechanism exists; `HEX_ID_FLOOR`/`HEX_SUPEREGO_FLOOR` go through the exact same code path as a hazard or an avatar.
- **Not yet done:** the vignette/lighting post-process (`art_and_ui.md §1`'s "spotlight in total darkness") — a PixiJS filter, unrelated to the sprite pipeline itself, still open. DOM-side chrome (tutorial box, legend panel, HUD) can get frames via CSS `border-image`/`background-image` — no engine change needed there at all, still open.
- **Verified:** `tsc --noEmit` clean, `npm run build` clean, `npx vitest run` 14/14 (unaffected — no test touches rendering), dev server boots cleanly and serves the two promoted `.webp` assets at their expected `SPRITE_PATHS` URLs.

## 4. The Generative Pipeline: Consistency Is the Real Problem

Generating *a* nice image is easy. Generating **fifty-plus images that all look like they belong to the same game** is the actual hard part, and it's a pipeline/process problem, not a prompting problem.

> **Tool decision reversed after live research (2026-07-21) — see §4.1.** The original recommendation (ComfyUI) assumed it was the more automatable option. Checking current sources changed that: **Draw Things**, not ComfyUI, is the better fit for this specific hardware. Full reasoning and sources below; this section documents *why*, since the space moves fast enough that the reasoning matters more than the conclusion — revisit if either tool changes materially.

### 4.1 Why Draw Things, not ComfyUI, on this hardware

ComfyUI's actual execution backend on macOS is PyTorch's MPS backend — a general CUDA-first codebase retrofitted onto Metal. As of live GitHub issues from the ComfyUI maintainers' own repo (dated into April 2026), this backend has real, currently-open problems: FP8 models crash outright (`Float8_e4m3fn` type errors), `--force-channels-last` throws runtime errors, `torch.compile` isn't available on MPS at all, and IC-LoRAs are CUDA-only. Workarounds exist but require hand-patching files that get overwritten on every ComfyUI update — not a foundation to build an unattended, repeatable asset pipeline on.

Draw Things is a native Swift/Metal application built specifically for Apple Silicon (not a Python/PyTorch port), and it has matured well past "simple GUI tool" since I last had solid information on it:
- **Performance:** its own engineering blog documents "Metal FlashAttention 2.0," and multiple independent sources report it running 20–40% faster than ComfyUI's MPS backend on identical Apple Silicon hardware, with on-demand weight loading cutting memory overhead by roughly half — directly relevant on a 36GB machine.
- **LoRA training is on-device and current**: Draw Things' own wiki and engineering blog confirm QLoRA-style training (quantized weights, low peak memory — their own SDXL example cites ~10GB peak) for SDXL, FLUX.1 [dev], Kwai Kolors, SD3.5, and — as of a recent update — **FLUX.2 [klein] (4B/9B), Qwen Image, and Z Image**. This is exactly the per-dimension style-LoRA technique §4.2 below relies on, and it runs on the hardware already in hand.
- **ControlNet + reference-image conditioning exist**: IP-Adapter Plus, IP-Adapter Plus Face, and "Redux" (image-to-style transfer) are built in.
- **Automation surface, several layers deep**: a local HTTP API (port 7860) for external control, a gRPC server with a "Server Assist" mode explicitly designed to offload generation to a second, more powerful Mac on the same network, a native CLI (`draw-things-cli`), and a sandboxed JavaScript scripting system for in-app workflow automation.
- **Confirmed hands-on (2026-07-21): `draw-things-cli` alone is enough — no HTTP server, no MCP bridge needed for the core pipeline.** Installed via `brew install --cask draw-things` + `brew install draw-things-cli`. `draw-things-cli generate --model <id> --prompt "..." --output out.png` runs fully headless, auto-downloads missing models, and accepts `--negative-prompt`, `--width`/`--height`, `--steps`, `--cfg`, `--seed`, `--image` (img2img/reference), and `--config-json`/`--config-file` for LoRA selection (`JSGenerationConfiguration` format) — everything the Skill in §5 needs, directly scriptable from Bash. The `mcp-drawthings` bridge (still worth knowing about) turns out to be unnecessary for this use case.
- **First real style test, same day:** `flux_2_klein_4b_q8p.ckpt`, 512×512, 6 steps, no LoRA yet — an Id hazard (organic/thorned door) and a Superego hazard (rusted vault door) both rendered on-brief in 16–24 seconds each, including model load. The Id/Superego material contrast (§1's whole visual thesis) is already clearly legible from prompting alone, before any LoRA or reference-image work — a strong early signal that the plan in §4.3 is sound. Test outputs: `artwork_tests/test_id_hazard_locked.png`, `artwork_tests/test_superego_hazard_locked.png` (gitignored scratch folder, not final assets).
- **The Mac Studio fits naturally later**: "Server Assist" offloading is precisely the "M3 Pro now, Studio next week" situation — the same app, pointed at a second machine's GPU, once it's available. No separate tool or workflow needed for that transition.

The honest trade-off: ComfyUI's node graph is still the more powerful tool in the abstract for very complex, multi-model pipelines, and its workflow JSON files are trivially scriptable in a way a GUI tool normally isn't. But that power sits on the weaker backend for this specific hardware. Since the hardware is fixed (Mac only, no CUDA GPU in the picture), the tool actually engineered for it wins. Nothing rules out adding ComfyUI later for one specific exotic technique that never gets a Draw Things equivalent — worth revisiting per-technique, not as a wholesale second migration.

### 4.2 Model choice

- **Primary: FLUX.2 [klein]** (Black Forest Labs, released January 2026). Apache-2.0 — genuinely free for commercial use, no fee, no per-seat license — and now has native on-device LoRA training support in Draw Things. Strong prompt adherence and material rendering (useful for "etched Bakelite," "frosted glass," the game's whole material-forward art direction), and fast enough (sub-second at low step counts in its distilled form) for quick iteration during style exploration.
- **Explicit licensing flag: avoid FLUX.2 [dev] for anything that ships.** Dev is the 32B open-weights tier Black Forest Labs offers for free non-commercial use, but commercial self-hosted use requires a **paid license (reported at $999/month)**. Since this game is presumably headed toward release, treat Dev as private-experimentation-only (or skip it) and keep production work on Klein.
- **Secondary/fallback: SDXL.** Still has, by a wide margin, the deepest LoRA/ControlNet ecosystem of any open model — useful if Klein ever struggles with a specific technique that the SDXL community has already solved. Cheap to keep both installed; there's no need to commit to only one.
- **Worth a quick look, not a commitment yet:** Qwen-Image (Apache-2.0, exceptionally good at rendering legible in-image text) — not obviously needed since the game itself is deliberately text-free, but could matter if the Monitor's CRT annotations (`narrative.md §5.2b` — the one place text exists in this game) ever move from DOM/CSS into baked illustration.
- Sit this model choice inside the style bible (§Phase 0), not as a one-time pick — re-check it if a new model becomes clearly better on both quality *and* Draw Things support before Phase 2 starts, since this field moves in months, not years.

### 4.3 Consistency technique, in order of effort vs. payoff

1. **A locked prompt template + fixed reference image set**, used as IP-Adapter conditioning on every generation — cheapest, works immediately, no training.
2. **A small per-dimension style LoRA** (one for the Id, one for the Superego, maybe a third for the shared Matrix/UI-chrome material language), trained on-device in Draw Things on 10–20 hand-picked/curated images that already match the style bible. This is the single highest-leverage step for keeping dozens of assets visually coherent — far more reliable than repeating prompt text alone, and per §4.1, Draw Things can now train this directly for FLUX.2 [klein].
3. **ControlNet** (edge/depth conditioning) where a new asset needs to match an existing composition exactly (e.g., a "powered" variant of an already-generated conduit plate) rather than being generated fresh each time.

**Where the Mac Studio (512GB) actually matters:** not for everyday inference — Klein/SDXL inference is comfortable on the M3 Pro already. It matters for LoRA training throughput and any later push into heavier animation work (§6). Practical sequencing stays the same as before: do the style exploration and prompt-template work now on the M3 Pro; either train the first style LoRA on the M3 Pro directly (Draw Things' QLoRA approach is light enough — ~10GB peak for an SDXL-class model per their own numbers) or queue it for the Studio once available via Server Assist — the style bible and curated reference set produced this week is exactly what that training run needs as input either way.

## 5. The Skill

**Status: the tool is installed and the base generation path is already proven (§4.1) — building the Skill itself is the next concrete step, not blocked on anything further.** Since `draw-things-cli` alone is sufficient (no HTTP server or MCP bridge needed), a Claude Code **Skill** (e.g. `generate-sprite`) can wrap it directly:

1. **Input:** an asset identifier (ideally the exact `SpriteId` name, e.g. `HAZARD_LOCKED_RED`) plus a short content description ("a fleshy sphincter door, sealed").
2. **Preset lookup:** the skill maps the identifier to the right style preset — Id-organic, Superego-clinical, or Matrix/neutral — and appends the locked prompt template, negative prompt, model/LoRA choice, and reference image(s) for that preset automatically (via `--config-json`/`--config-file` for LoRA, `--image` for reference conditioning), so the calling side never has to remember or restate the style rules.
3. **Generation call:** `draw-things-cli generate --model flux_2_klein_4b_q8p.ckpt --prompt "<template + content>" --negative-prompt "..." --width … --height … --steps … --seed … --output <path>` — a single Bash invocation, no server process to keep alive.
4. **Post-processing:** crop/pad to the icon canvas size, force a transparent background for foreground sprites (vs. a full-bleed export for backgrounds/panels), export to WebP/PNG at the path `SPRITE_PATHS` already expects.
5. **Selection:** generate a small batch (e.g. 4, varying only `--seed`) per asset and present them (as an Artifact or a quick contact sheet) for a pick, since diffusion output quality varies run to run — this keeps a human in the loop for the one judgment call that actually matters (does this look right?) without hand-holding the mechanical steps.

This needs no cloud access and no Anthropic-side image tool — it's a local process Claude Code drives the same way it drives any other CLI tool via Bash.

## 6. Animation — realistic expectations

Two very different things are both called "animation" here, worth separating:

- **Procedural "aliveness"** (recommended first, and mostly free): pulsing glow on powered matrix nodes, a slow breathing/bob on the avatars, a flicker on hazards — all achievable today with `TweenManager.ts` (already exists) or a simple shader uniform, using a single static image per asset. No generative-AI animation work needed at all, fully consistent by construction (it's math, not more diffusion output), and it's most of what makes a static scene feel like it's part of a living diorama rather than a photo of one.
- **True generated frame animation** (a real `AnimatedSprite` frame sequence) is the harder ask: diffusion models are notoriously bad at frame-to-frame consistency (flicker, drifting proportions) without dedicated tooling (AnimateDiff-style workflows, img2img loopback chains, or a proper video model). This is worth doing selectively, later, for a couple of genuine "hero" moments (the avatar idle, a powered-node pulse if the procedural version isn't enough) rather than across the whole asset set, and it's exactly the kind of heavier, iterate-many-times workload that's much more comfortable on the Mac Studio than the M3 Pro.
- The old Threshold board-flip animation described in `art_and_ui.md §6` should not be built — the mechanic is gone (SPRINT_026).

- **Second real fix needed the same day, caught by Till looking at the actual output, not by any automated check:** the first promoted pair had the model draw its own "hexagonal tile on black" — which came out as an octagon, and worse, cropping that with the code-side hex mask let the octagon's own black background show through as wedges inside the hex frame ("an octagon inside a hexagon"). Root cause: asking the model to handle the silhouette at all was the mistake, not a prompt-wording problem to iterate on. Fix: prompt for a full-bleed square texture with content to every edge (no shape language, no black background at all in the source) and let the hex mask alone define the silhouette, cutting clean hex content in every direction since there's no black in the source to leak through. See `generate-sprite/SKILL.md`'s floor-tile framing clause for the corrected wording.
- **Third round of fixes, this time from Till actually looking at it running in the browser** — the one check I explicitly flagged I couldn't do myself (no Playwright in this project). Two real problems surfaced that no amount of build/test verification would have caught:
  1. **Resolution/detail mismatch.** The source art (512–1024px, fine photographic detail — individual veins, tiny bone fragments, individual scratches) was tuned for a display size far larger than the ~80px a hex actually renders at (`SPRITE_SIZE = HEX_SIZE*2` in `RenderSystem.ts`). Downscaled that hard, fine detail just becomes visual mush. Fix, two parts: (a) `postprocess.py` gained `--pixelate N` — downscale to an N×N grid (BOX filter, a clean area-average) then back up with NEAREST (hard edges, no re-blur) — a deliberate low-fi/pixel-art look instead of a photoreal texture fighting the display size; (b) `PixiDriver.ts` now sets `texture.source.scaleMode = 'nearest'` on every loaded texture, so PixiJS doesn't quietly re-smooth the deliberately blocky pixels back into blur at render time. **Pixelating an already-detailed source just turned it to colorful noise, not clean pixel art** — see next point, the two problems compound.
  2. **"Matsch aus immer gleichem Muster"** — the base tile content itself was too busy (dense competing veins/bones/scratches everywhere) for a texture meant to repeat across many tiles. Regenerated both dimensions with a deliberately **uniform, low-detail base material** prompt (plain mottled velvet / brushed steel with only subtle variation, explicit negatives against "distinct objects"/"fine detail"/"clutter") — this is what `--pixelate` actually needed to produce clean blocky regions instead of noise. Distinct props (a bone, a vein, a rivet) are deferred to sparse occasional overlays (not yet built — see the open item below), matching Till's own description: "eine einheitliche Landschaft mit gelegentlichen Artefakten."
- **New: a persistent hex border + hover highlight**, added because flat photographic art has no natural tile edge the way a flat color fill did ("es ist nicht ganz klar, wo die Felder enden"). `RenderCommandBuffer.ts` gained `DrawHexBorderCommand`; `RenderSystem.ts`'s floor pass now emits a permanent low-contrast "grout" outline on every tile plus a bright highlight on whichever hex the pointer is over. Hover tracking is new (`uiState.hoveredHex`, populated by a `mousemove` listener in `MouseInput.ts` reusing the exact same coordinate-mapping the click-to-move handler already had) — this is a small new interaction feature, not just a rendering tweak, worth being aware of if hover ever needs to extend to other purposes (tooltips, etc.).
- **Fourth round, from Till running it in the browser again:** even the uniform/pixelated single texture still reads as "the same pattern repeated" once tiled across many hexes ("wesentlich zu viel wiedererkennbares Muster") — confirming that no single texture, however uniform, escapes looking stamped once it's the same exact image dozens of times. Till also flagged that the Superego side is the opposite case: an obviously-repeating pattern is *fine* there thematically (clinical/mechanical), so it only got the pixelation fix, not the multi-variant one. Two structural additions landed the same day:
  1. **Multiple tile variants, picked per-hex by a stable hash.** `PixiDriver.ts`'s texture map became `Map<SpriteId, Texture[]>`; it now probes for numbered variant files (`hex_id_floor.webp`, `hex_id_floor_2.webp`, `hex_id_floor_3.webp`, ...) at startup. `RenderSystem.ts` computes a cheap hash of each hex's `(q, r)` and uses it to pick a variant — stable across frames/reloads (the same hex always shows the same variant) but different neighboring hexes usually differ. `DrawSpriteCommand` gained an optional `variantSeed`. Four Id variants promoted (`hex_id_floor.webp` + `_2`/`_3`/`_4`); Superego intentionally stays at one.
  2. **A cosmetic multi-hex decal overlay, randomly scattered fresh on every level load** — Till's own proposed fix for repetition not fully solved by variants alone: larger props (Id: a bone/vein cluster; Superego: a rivet row) spanning roughly 2.5 hex-widths, transparent background, same pixelated house style, layered on top of the floor tiles at 3 random positions per dimension per load (`state/DecalState.ts`, a new `DrawDecalCommand`, `PixiDriver.ts` given its own `decalLayer`/pool/texture map keyed by asset path rather than `SpriteId` since decals aren't ECS entities). Deliberately **not networked and not derived from any shared seed** — cosmetic-only, so Host and Guest scattering independently and disagreeing is fine, unlike everything else in this game. One decal per dimension exists so far (`decal_id_1.webp`, `decal_superego_1.webp`); more variety, rotation/flip randomization, and density tuning are open — this is a first working prototype of the concept, not the finished feature.
- **Hex border color also corrected the same round:** explicitly **not black or white** (Till's instruction) — a neutral warm gray (`0x4A4038`), low alpha (0.28) but thicker (4px, was 1px) than the first attempt, so it reads as a rough seam rather than either a stark grid line or nothing at all.
- **Fifth round, from Till again actually running it:**
  1. **The Superego base tile read as "a lamp cluster."** Root cause: the texture had its own per-tile radial highlight (bright center, dark edges — a directional-gloss look), which repeats into what looks like a grid of individually-lit bulbs/orbs rather than one continuous floor. Fixed by re-prompting for **completely flat, uniformly-lit** shading ("no directional light," "no highlight," "no gloss," "no hot spot" in the negative prompt) — took two attempts to fully kill the residual bright patch; the working version reads as pale sterile ceramic/morgue-floor tile with no gloss at all. The Id side didn't have this problem and needed no change ("looks very good already").
